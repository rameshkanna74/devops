Imagine a receptionist.

Instead

```text

↓

↓

```

Nginx does exactly this.

# 3. Networking Concepts

Today we learn five new concepts.

---

## Concept 1

## Client vs Reverse Proxy

The browser connects only to Nginx.

```text
Browser

↓

Nginx
```

Nginx becomes the HTTP client for FastAPI.

```text
Nginx

↓

FastAPI
```

One request becomes two separate HTTP connections.

---

## Packet Flow

```text
Browser

↓

TCP

↓

Nginx

↓

HTTP Proxy

↓

Docker DNS

↓

backend

↓

FastAPI

↓

Response

↓

Nginx

↓

Browser
```

Notice something important.

The browser never communicates directly with FastAPI.

---

## Concept 2

## Docker DNS

Level 1 taught us

```python
http://server:8000
```

Now Nginx will do the same.

Inside nginx.conf

```nginx
proxy_pass http://backend:8000;
```

Notice

No IP.

No localhost.

No static configuration.

Docker DNS.


# backend/app.py

```python
from fastapi import FastAPI
from datetime import datetime
import socket

app = FastAPI()

@app.get("/")
def home():

    return {

        "message": "Hello from FastAPI",

        "hostname": socket.gethostname(),

        "time": str(datetime.now())

    }
```

Why include the hostname?

Later we'll add multiple backend containers.

Seeing the hostname lets us identify which container handled the request.

This is a common technique when learning load balancing.

# nginx.conf

```nginx
events {}

http {

    server {

        listen 80;

        location / {

            proxy_pass http://backend:8000;

        }

    }

}
```

This tiny file is doing something very powerful.


# Verify Docker Networks

List networks

```bash
docker network ls
```

You'll see a default bridge network created by Compose.

Inspect it

```bash
docker network inspect <project>_default
```

Observe

* Nginx attached
* Backend attached
* Individual container IPs
* Network aliases

Pay particular attention to the aliases. You should see `backend`, which is the DNS name Nginx uses in `proxy_pass`.

---

# Verify Backend Is Private

Try to access the backend directly from your host:

```bash
curl http://localhost:8000
```

You should receive a connection error because the backend no longer publishes a host port.

This demonstrates an important security principle: the application is reachable only through the reverse proxy.

---

# Verify Connectivity from Inside Nginx

Open a shell:

```bash
docker compose exec nginx sh
```

Install a simple HTTP client if needed (the Alpine image is minimal):

```bash
apk add --no-cache curl
```

Now test the backend directly:

```bash
curl http://backend:8000
```

You should receive the same JSON response.

This proves that:

1. Docker DNS resolves `backend`.
2. The backend is reachable over the internal bridge network.
3. Nginx and the backend share the same Docker network.

# Key Takeaways (Part 1)

You have transformed your architecture from directly exposing the application to using a dedicated reverse proxy.

You also learned three production practices that you'll keep using throughout the remaining levels:

* Only expose the reverse proxy to the outside world.
* Keep backend services private on the Docker network.
* Use Docker service names instead of IP addresses for inter-container communication.
