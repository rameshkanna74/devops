Almost every production application looks like this:

```

↓

↓

↓
```

Every arrow is network communication.

If networking breaks...

# 2. Networking Concepts

Today we only learn four concepts.

Nothing more.

---

## Concept 1 — Every Container is its Own Computer

Imagine your laptop.

It has:

* IP address
* Network interface
* Hostname
* DNS
* Routing table

A Docker container has the same things.

```
+-------------------+
| Container A       |
|-------------------|
| IP Address        |
| Hostname          |
| DNS               |
| Routes            |
+-------------------+

+-------------------+
| Container B       |
|-------------------|
| IP Address        |
| Hostname          |
| DNS               |
| Routes            |
+-------------------+
```

Containers are isolated.

---

## Concept 2 — localhost

This is the single biggest beginner mistake.

Suppose we have:

```
client

server
```

The client tries:

```
http://localhost:8000
```

What happens?

```
Client Container

localhost

↓

Client itself
```

NOT

```
Server
```

Remember:

> localhost always means **ME**

Never another container.

---

## Concept 3 — Docker Network

Containers join a virtual switch.

```
        Docker Bridge Network

      +-----------------------+

 Client               Server

      +-----------------------+
```

Every container attached to the same bridge can communicate.

---

## Concept 4 — Docker DNS

Docker automatically creates DNS entries.

Suppose the service is named:

```
server
```

Then inside another container:

```
http://server:8000
```

works immediately.

No IP addresses.

No configuration.

Docker DNS handles it.

This is why production Docker Compose files almost never hardcode container IP addresses.

---

# Packet Flow

When the client calls:

```
http://server:8000
```

```
Client

↓

DNS lookup

↓

Docker DNS

↓

server

↓

Container IP

↓

TCP connection

↓

FastAPI
```

That entire process happens inside the Docker network.

# 5. Build Everything

## `docker-compose.yml`

```yaml
version: "3.9"

services:
  server:
    build: ./server
    container_name: server
    ports:
      - "8000:8000"

  client:
    build: ./client
    container_name: client
    depends_on:
      - server
```

### Docker Compose Features Introduced

#### `build`

**What it does:** Builds an image from the specified directory.

**Why it exists:** So you can define how to package your application.

**Common mistake:** Forgetting the Dockerfile in the target directory.

**Interview expectation:** Know the difference between `build:` and using a prebuilt image with `image:`.

---

#### `container_name`

**What it does:** Assigns a fixed name to the container.

**Why it exists:** Makes it easier to inspect and debug.

**Common mistake:** Reusing the same name across projects causes conflicts.

**Interview expectation:** Explain that Compose service names already provide DNS names; `container_name` is often unnecessary in larger Compose projects.

---

#### `ports`

**What it does:** Publishes a container port to the host.

```
Host Port : Container Port

8000 : 8000
```

**Why it exists:** Allows your Windows browser or `curl` from WSL to access the server.

**Common mistake:** Assuming publishing is required for container-to-container communication. It is not.

**Interview expectation:** Distinguish between internal container networking and host port publishing.

---

#### `depends_on`

**What it does:** Starts the `server` service before the `client`.

**Why it exists:** Helps with startup order.

**Common mistake:** Thinking it waits until the server is ready. It only controls start order, not application readiness.

**Interview expectation:** Mention that health checks are needed for readiness, which will be covered in a later level.

# 6. Verification

Build and start the project:

```bash
docker compose up --build
```

### Verify containers

```bash
docker compose ps
```

Observe that both containers are running and the server has port `8000` published to the host.

### Check logs

```bash
docker compose logs server
docker compose logs client
```

You should see the client print:

```text
{'message': 'Hello from the server!'}
```

### Verify from your host

From WSL:

```bash
curl http://localhost:8000
```

Expected:

```json
{"message":"Hello from the server!"}
```

**Why `localhost` works here:** The server's port is published from the container to your host. Your browser or WSL talks to the host, which forwards the traffic into the container.

---

## Inspect networks

```bash
docker network ls
```

Find the automatically created Compose bridge network (typically named `<project>_default`).

Inspect it:

```bash
docker network inspect <network_name>
```

Observe:

* Network driver (`bridge`)
* Connected containers
* Assigned IP addresses
* Aliases (including service names)

**Why this matters:** In production, this command quickly shows whether services are on the same network.

---

## Enter the client container

```bash
docker compose exec client sh
```

Inside the container:

```bash
hostname
```

Displays the container hostname.

Check DNS resolution:

```bash
getent hosts server
```

This resolves `server` to its container IP using Docker's embedded DNS. If `getent` isn't available, we'll install additional networking tools in the next level.

# 7. Break the System

Do **not** fix these yet. Treat them like production incidents.

1. Change the client URL to:

```python
http://localhost:8000
```

2. Rebuild:

```bash
docker compose up --build
```

Question:

**What do you think will happen? Why?**

---

3. Change the hostname in the client to:

```python
http://backend:8000
```

Question:

**What error do you expect?**

---

4. Remove the `ports:` section from the server.

Question:

* Will the client still be able to reach the server?
* Will your browser still be able to reach the server?

Explain your reasoning before testing.

# 8. Debugging Workflow

When one of the above failures occurs, don't jump straight to the fix.

Follow this order:

1. What exactly is the error? (`Connection refused`, `Name or service not known`, `Timeout`, etc.)
2. Is the container running?
3. Are both containers on the same Docker network?
4. Does the service name resolve through Docker DNS?
5. Is the application listening on the expected port?
6. Is the URL using the correct hostname?

This systematic approach is what interviewers look for.

# 9. Linux Networking (Level 1)

Commands to know:

```bash
hostname
```

Shows the current container's hostname.

```bash
ip addr
```

Shows network interfaces and IP addresses (install `iproute2` if your image doesn't include it).

```bash
ip route
```

Displays the routing table.

```bash
curl http://server:8000
```

Tests HTTP connectivity from inside a container.

These tools become your first line of investigation during outages.

# 14. Cheat Sheet


### Mental Models

* `localhost` = **this container**
* Service name = Docker DNS hostname
* Published ports are for **host ↔ container** communication
* Shared bridge network enables **container ↔ container** communication

### Common Mistakes

* Using `localhost` between containers
* Hardcoding container IP addresses
* Confusing published ports with internal networking
* Assuming `depends_on` waits for application readiness
